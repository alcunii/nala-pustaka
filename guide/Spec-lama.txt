Requirements Document
Introduction
Sistem RAG (Retrieval-Augmented Generation) untuk Naskah Babad adalah sebuah platform AI yang memungkinkan pengguna untuk melakukan pencarian semantik dan berinteraksi dengan koleksi naskah Jawa kuno melalui chatbot AI. Sistem ini akan memproses 121+ file naskah babad, mengindeksnya ke dalam vector database, dan menyediakan interface untuk query berbasis natural language dalam bahasa Indonesia dan Jawa.
Glossary
⦁	RAG System: Sistem yang menggabungkan retrieval (pencarian) dengan generation (pembangkitan teks) menggunakan AI untuk menjawab pertanyaan berdasarkan dokumen yang tersimpan
⦁	Vector Database: Database yang menyimpan representasi numerik (embeddings) dari teks untuk pencarian semantik
⦁	Embedding: Representasi numerik dari teks yang menangkap makna semantiknya
⦁	Semantic Search: Pencarian berdasarkan makna/konteks, bukan hanya keyword matching
⦁	Chunking: Proses memecah dokumen besar menjadi bagian-bagian kecil untuk diproses
⦁	Web Application: Aplikasi berbasis web yang dapat diakses melalui browser
⦁	API Endpoint: Interface untuk komunikasi antara frontend dan backend
⦁	Naskah Babad: Koleksi naskah sejarah Jawa yang tersimpan dalam format teks
Requirements
Requirement 1: Data Processing and Ingestion
User Story: Sebagai administrator sistem, saya ingin dapat memproses dan mengindeks semua naskah babad ke dalam vector database, sehingga naskah-naskah tersebut dapat dicari secara semantik oleh pengguna.
Acceptance Criteria
1.	WHEN administrator menjalankan proses ingestion, THE RAG System SHALL membaca semua file teks dari direktori data_naskah/naskah_babad
2.	WHEN sebuah file naskah dibaca, THE RAG System SHALL mengekstrak metadata (judul, penulis, tahun, URL, nomor ID) dari header file
3.	WHEN teks naskah diproses, THE RAG System SHALL memecah teks menjadi chunks dengan ukuran maksimal 1000 tokens dengan overlap 200 tokens
4.	WHEN chunks dibuat, THE RAG System SHALL mempertahankan informasi metadata dan nomor halaman untuk setiap chunk
5.	WHEN chunks siap, THE RAG System SHALL menghasilkan embeddings menggunakan model embedding multilingual yang mendukung bahasa Jawa dan Indonesia
Requirement 2: Vector Database Storage
User Story: Sebagai sistem, saya ingin menyimpan embeddings dan metadata naskah dalam vector database yang efisien, sehingga pencarian semantik dapat dilakukan dengan cepat dan akurat.
Acceptance Criteria
1.	THE RAG System SHALL menyimpan embeddings dalam vector database dengan dukungan similarity search
2.	WHEN menyimpan embedding, THE RAG System SHALL menyertakan metadata lengkap (judul naskah, penulis, tahun, URL, nomor halaman, chunk text)
3.	THE RAG System SHALL mendukung indexing untuk minimal 121 naskah dengan total estimasi 50,000+ chunks
4.	WHEN database diakses, THE RAG System SHALL mengembalikan hasil dalam waktu kurang dari 2 detik untuk query standar
5.	THE RAG System SHALL menyediakan mekanisme untuk update dan delete embeddings ketika naskah diperbarui
Requirement 3: Semantic Search Functionality
User Story: Sebagai pengguna, saya ingin dapat mencari informasi dalam naskah menggunakan bahasa natural (Indonesia atau Jawa), sehingga saya dapat menemukan konten yang relevan tanpa harus tahu keyword exact.
Acceptance Criteria
1.	WHEN pengguna mengirim query, THE RAG System SHALL mengkonversi query menjadi embedding menggunakan model yang sama dengan indexing
2.	WHEN embedding query dibuat, THE RAG System SHALL melakukan similarity search di vector database
3.	WHEN similarity search dilakukan, THE RAG System SHALL mengembalikan top 5-10 chunks paling relevan dengan similarity score minimal 0.7
4.	WHEN hasil ditemukan, THE RAG System SHALL mengurutkan hasil berdasarkan relevance score dari tertinggi ke terendah
5.	THE RAG System SHALL mengembalikan hasil dengan metadata lengkap termasuk judul naskah, penulis, tahun, dan nomor halaman
Requirement 4: AI Chatbot Interface
User Story: Sebagai pengguna, saya ingin dapat bertanya kepada AI tentang isi naskah dalam bahasa natural, sehingga saya mendapatkan jawaban yang kontekstual dan mudah dipahami.
Acceptance Criteria
1.	WHEN pengguna mengirim pertanyaan, THE RAG System SHALL melakukan semantic search untuk menemukan chunks relevan
2.	WHEN chunks relevan ditemukan, THE RAG System SHALL mengirim chunks tersebut sebagai context ke LLM (Large Language Model)
3.	WHEN LLM menerima context, THE RAG System SHALL menghasilkan jawaban yang koheren dalam bahasa yang sama dengan pertanyaan
4.	WHEN jawaban dihasilkan, THE RAG System SHALL menyertakan sumber referensi (judul naskah, halaman) untuk setiap informasi
5.	THE RAG System SHALL mendukung conversation history untuk follow-up questions dalam satu sesi chat
Requirement 5: Web Application Frontend
User Story: Sebagai pengguna, saya ingin mengakses sistem RAG melalui web interface yang user-friendly, sehingga saya dapat dengan mudah mencari dan bertanya tentang naskah.
Acceptance Criteria
1.	THE Web Application SHALL menyediakan search bar untuk input query atau pertanyaan
2.	WHEN pengguna melakukan pencarian, THE Web Application SHALL menampilkan hasil dalam format yang mudah dibaca dengan highlighting
3.	THE Web Application SHALL menyediakan chat interface untuk interaksi dengan AI chatbot
4.	WHEN hasil ditampilkan, THE Web Application SHALL menampilkan metadata naskah (judul, penulis, tahun) dan link ke sumber asli
5.	THE Web Application SHALL responsive dan dapat diakses dari desktop dan mobile devices
Requirement 6: API Backend Services
User Story: Sebagai developer, saya ingin memiliki API backend yang terstruktur dengan baik, sehingga frontend dapat berkomunikasi dengan sistem RAG secara efisien dan aman.
Acceptance Criteria
1.	THE RAG System SHALL menyediakan REST API endpoint untuk semantic search dengan method POST
2.	THE RAG System SHALL menyediakan REST API endpoint untuk chat completion dengan method POST
3.	WHEN API menerima request, THE RAG System SHALL memvalidasi input dan mengembalikan error message yang jelas jika invalid
4.	WHEN API memproses request, THE RAG System SHALL mengembalikan response dalam format JSON dengan struktur yang konsisten
5.	THE RAG System SHALL mengimplementasikan rate limiting untuk mencegah abuse dengan maksimal 60 requests per menit per user
Requirement 7: Model Selection and Configuration
User Story: Sebagai administrator, saya ingin dapat mengkonfigurasi model AI yang digunakan untuk embedding dan generation, sehingga sistem dapat dioptimalkan untuk performa dan biaya.
Acceptance Criteria
1.	THE RAG System SHALL mendukung konfigurasi embedding model melalui environment variables atau config file
2.	THE RAG System SHALL mendukung multiple LLM providers (OpenAI, Google Gemini, Anthropic Claude)
3.	WHEN model dikonfigurasi, THE RAG System SHALL memvalidasi API keys dan koneksi sebelum memulai service
4.	THE RAG System SHALL menyimpan konfigurasi model dalam file terpisah dari kode aplikasi
5.	WHERE administrator memilih model lokal, THE RAG System SHALL mendukung model open-source seperti sentence-transformers untuk embedding
Requirement 8: Data Privacy and Security
User Story: Sebagai administrator, saya ingin memastikan data naskah dan query pengguna aman, sehingga privasi dan integritas data terjaga.
Acceptance Criteria
1.	THE RAG System SHALL mengenkripsi API keys dan credentials dalam environment variables
2.	WHEN data dikirim ke external API, THE RAG System SHALL menggunakan HTTPS protocol
3.	THE RAG System SHALL tidak menyimpan query pengguna secara permanen kecuali untuk logging dengan consent
4.	WHERE logging diaktifkan, THE RAG System SHALL mengimplementasikan log rotation dengan retention maksimal 30 hari
5.	THE RAG System SHALL menyediakan opsi untuk self-hosted deployment tanpa ketergantungan pada external API
Requirement 9: Performance and Scalability
User Story: Sebagai pengguna, saya ingin sistem merespons dengan cepat bahkan ketika banyak pengguna mengakses secara bersamaan, sehingga pengalaman pengguna tetap optimal.
Acceptance Criteria
1.	WHEN pengguna melakukan semantic search, THE RAG System SHALL mengembalikan hasil dalam waktu kurang dari 2 detik
2.	WHEN pengguna bertanya ke chatbot, THE RAG System SHALL mulai streaming response dalam waktu kurang dari 3 detik
3.	THE RAG System SHALL mendukung minimal 10 concurrent users tanpa degradasi performa signifikan
4.	WHERE vector database size melebihi 100,000 chunks, THE RAG System SHALL tetap mempertahankan response time di bawah 3 detik
5.	THE RAG System SHALL mengimplementasikan caching untuk queries yang sering diakses dengan TTL 1 jam
Requirement 10: Monitoring and Maintenance
User Story: Sebagai administrator, saya ingin dapat memonitor kesehatan sistem dan melakukan maintenance, sehingga sistem dapat berjalan stabil dan issue dapat dideteksi lebih awal.
Acceptance Criteria
1.	THE RAG System SHALL menyediakan health check endpoint yang mengembalikan status database dan API connections
2.	WHEN error terjadi, THE RAG System SHALL mencatat error details dalam log file dengan timestamp dan context
3.	THE RAG System SHALL menyediakan metrics endpoint untuk monitoring jumlah queries, response time, dan error rate
4.	THE RAG System SHALL menyediakan script untuk re-indexing naskah ketika ada update atau perubahan model
5.	WHERE sistem mengalami downtime, THE RAG System SHALL mengembalikan error message yang informatif kepada pengguna

# Design Document: RAG System untuk Naskah Babad

## Overview

Sistem RAG (Retrieval-Augmented Generation) untuk Naskah Babad adalah upgrade dari implementasi RAG sederhana yang sudah ada di Nala Pustaka. Sistem baru ini akan mengimplementasikan vector database untuk semantic search yang lebih akurat, chunking strategy yang optimal untuk naskah Jawa kuno, dan API backend yang scalable.

### Current State
- Frontend: React 18 + Vite + Tailwind CSS
- Database: Supabase (PostgreSQL)
- AI: Google Gemini 2.0 Flash (basic RAG dengan full text)
- Limitation: Mengirim full text naskah setiap query (tidak efisien, token limit)

### Target State
- Vector Database: Pinecone / Qdrant / Chroma (untuk semantic search)
- Embedding Model: sentence-transformers (multilingual support)
- Backend API: Node.js + Express / Python + FastAPI
- LLM: Google Gemini 2.5 Flash Lite (higher rate limits, cost-effective)
- Advanced RAG: Chunking + Embedding + Similarity Search + Context Augmentation
- Deployment: Vercel (frontend) + Railway/Render (backend)

### Component Interaction Flow

**1. Data Ingestion Flow (One-time / Update)**
```
Raw Text Files → Text Processor → Chunker → Embedding Generator → Vector DB
                                                    ↓
                                            Metadata Storage (Supabase)
```

**2. Semantic Search Flow**
```
User Query → Embedding Generator → Vector DB (Similarity Search) → Top K Chunks → Response
```

**3. Chat Flow (RAG)**
```
User Question → Embedding → Vector DB → Retrieve Relevant Chunks → 
Augment Prompt → LLM (Gemini) → Grounded Answer → User
```

## Data Models

### 1. Manuscript Model (Supabase)
```sql
CREATE TABLE manuscripts (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  title TEXT NOT NULL,
  author TEXT,
  year TEXT,
  description TEXT,
  url TEXT,
  full_text TEXT,
  is_pinned BOOLEAN DEFAULT FALSE,
  knowledge_graph JSONB,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  indexed_at TIMESTAMP,
  total_chunks INTEGER DEFAULT 0
);
```

### 2. Chunk Model (Metadata in Supabase)
```sql
CREATE TABLE chunks (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  manuscript_id UUID REFERENCES manuscripts(id) ON DELETE CASCADE,
  chunk_index INTEGER NOT NULL,
  text TEXT NOT NULL,
  page_number TEXT,
  vector_id TEXT UNIQUE,  -- ID in vector database
  created_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(manuscript_id, chunk_index)
);

CREATE INDEX idx_chunks_manuscript ON chunks(manuscript_id);
CREATE INDEX idx_chunks_vector ON chunks(vector_id);
```

### 3. Query Log Model (Optional - Analytics)
```sql
CREATE TABLE query_logs (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  query TEXT NOT NULL,
  query_type TEXT CHECK (query_type IN ('search', 'chat')),
  manuscript_id UUID REFERENCES manuscripts(id),
  response_time INTEGER,  -- milliseconds
  results_count INTEGER,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_query_logs_created ON query_logs(created_at);
```

## Error Handling

### Error Types and Responses

```typescript
enum ErrorCode {
  INVALID_INPUT = 'INVALID_INPUT',
  VECTOR_DB_ERROR = 'VECTOR_DB_ERROR',
  EMBEDDING_ERROR = 'EMBEDDING_ERROR',
  LLM_ERROR = 'LLM_ERROR',
  DATABASE_ERROR = 'DATABASE_ERROR',
  RATE_LIMIT_EXCEEDED = 'RATE_LIMIT_EXCEEDED',
  INTERNAL_ERROR = 'INTERNAL_ERROR'
}

interface ErrorResponse {
  error: {
    code: ErrorCode;
    message: string;
    details?: any;
  };
  timestamp: Date;
}
```

### Error Handling Strategy

1. **Input Validation Errors** (400)
   - Empty query
   - Invalid manuscript ID
   - Invalid parameters

2. **External Service Errors** (502/503)
   - Vector DB timeout
   - LLM API failure
   - Embedding service down
   - Retry with exponential backoff (max 3 attempts)

3. **Rate Limiting** (429)
   - 60 requests per minute per IP
   - Return Retry-After header

4. **Internal Errors** (500)
   - Log to monitoring service
   - Return generic error message
   - Alert admin

## Alternative Approaches Considered

### 1. Vector Database Options

**Pinecone (Chosen)**
- ✅ Managed service (no ops)
- ✅ Fast and scalable
- ✅ Good documentation
- ❌ Cost (free tier: 1 index, 100K vectors)

**Qdrant**
- ✅ Open source
- ✅ Self-hostable
- ✅ No vendor lock-in
- ❌ Requires ops/maintenance

**Chroma**
- ✅ Simple API
- ✅ Good for prototyping
- ❌ Less mature for production

**Decision**: Start with Pinecone for ease of use, provide Qdrant option for self-hosting.

### 2. Embedding Model Options

**sentence-transformers/paraphrase-multilingual-mpnet-base-v2 (Chosen)**
- ✅ Multilingual (Indonesian support)
- ✅ Good performance
- ✅ 768 dimensions
- ❌ Limited Javanese support

**OpenAI text-embedding-ada-002**
- ✅ High quality
- ✅ 1536 dimensions
- ❌ Cost per request
- ❌ External API dependency

**Decision**: Use sentence-transformers for cost and control, with option to switch to OpenAI.

### 3. Backend Framework Options

**Express.js (Chosen)**
- ✅ Familiar to team
- ✅ Large ecosystem
- ✅ Good for REST APIs
- ❌ Less opinionated

**FastAPI (Python)**
- ✅ Great for ML/AI services
- ✅ Auto-generated docs
- ✅ Type hints
- ❌ Different language from frontend

**Decision**: Express.js for consistency with frontend stack, FastAPI as alternative for ML-heavy workloads.

## Gemini 2.5 Flash Lite Integration Details

### Why Gemini 2.5 Flash Lite?

**Advantages for RAG System**:
1. **Higher Rate Limits**: 4000 RPM vs 2000 RPM (2x throughput)
2. **Cost-Effective**: Lower cost per request, ideal for high-volume queries
3. **Large Context Window**: 1M tokens (can handle multiple long chunks)
4. **Fast Response**: Optimized for low latency
5. **Good Quality**: Sufficient for grounded RAG responses

**Use Cases**:
- Semantic search with many concurrent users
- Chat interface with high query volume
- Batch processing for analytics
- Cost-sensitive deployments

### API Configuration

**Option 1: Google AI Studio (Simpler)**
```typescript
const apiEndpoint = 'https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash-lite:generateContent';
const apiKey = process.env.GEMINI_API_KEY;

// Request format (same as current implementation)
const payload = {
  contents: [{ parts: [{ text: prompt }] }],
  systemInstruction: { parts: [{ text: systemPrompt }] },
  generationConfig: {
    temperature: 0.3,
    maxOutputTokens: 1024,
    topP: 0.8,
    topK: 40
  }
};
```

**Option 2: Vertex AI (Production-grade)**
```typescript
// Requires Google Cloud Project + Service Account
const { VertexAI } = require('@google-cloud/vertexai');

const vertexAI = new VertexAI({
  project: process.env.GCP_PROJECT_ID,
  location: 'us-central1'
});

const model = vertexAI.getGenerativeModel({
  model: 'gemini-2.5-flash-lite'
});

const result = await model.generateContent({
  contents: [{ role: 'user', parts: [{ text: prompt }] }],
  systemInstruction: { parts: [{ text: systemPrompt }] },
  generationConfig: {
    temperature: 0.3,
    maxOutputTokens: 1024
  }
});
```

### Rate Limiting Strategy

With 4000 RPM, we can support:
- **66 requests per second**
- **~100 concurrent users** (assuming 1 query per user per 1.5 seconds)

**Implementation**:
```typescript
// Backend rate limiter
import rateLimit from 'express-rate-limit';

const chatLimiter = rateLimit({
  windowMs: 60 * 1000,  // 1 minute
  max: 60,  // 60 requests per minute per IP
  message: 'Terlalu banyak permintaan, coba lagi dalam 1 menit'
});

app.post('/api/chat', chatLimiter, chatHandler);
```

### Cost Estimation

**Gemini 2.5 Flash Lite Pricing** (as of documentation):
- Input: Lower cost per 1M tokens
- Output: Lower cost per 1M tokens
- Free tier: Available with API key from AI Studio

**Estimated Monthly Cost** (1000 users, 10 queries/user/month):
- Total queries: 10,000
- Avg input tokens: 2000 (context + query)
- Avg output tokens: 500
- **Estimated cost**: Significantly lower than standard models

**Current**: Gemini 2.0 Flash Experimental
**Target**: Gemini 2.5 Flash Lite
